1. EXECUTIVE SUMMARY & ARCHITECTURE OVERVIEW
================================================================================

The LD-ToyPad Bridge intercepts LEGO Dimensions' cellUsbd USB calls by:
  1. Installing 4-instruction preambles (lis/ori/mtctr/bctr) into the game's
     R-X .text segment via PS3MAPI Ring 0 memory writes.
  2. Each preamble branches to a 64-byte R-W-X trampoline (generated by
     trampoline_gen.c at runtime) that saves the game's TOC, loads the SPRX's
     TOC, calls the C hook via bctrl, restores the game's TOC, and returns.
  3. C hooks (in usb_hooks.c) intercept cellUsbdInit, cellUsbdOpenPipe,
     cellUsbdInterruptTransfer, and cellUsbdClosePipe. Toy Pad traffic is
     routed over UDP to the PC bridge server. Non-ToyPad traffic passes
     through to the real cellUsbd functions via direct calls.

The architecture is architecturally sound and the PowerPC trampoline code in
trampoline_gen.c is correct. HOWEVER, there is a CRITICAL mismatch between
the SPRX's IPC file output and the inject-sprx.js preamble writer that will
cause the hooks to NOT be installed — the PS3 will NOT crash, but the Toy
Pad traffic will never reach the PC server.

Additionally, the passthrough path for non-ToyPad USB devices is incomplete:
the current hooks call real cellUsbd functions directly (SPRX's own imports),
but if the game called them via the trampoline, the extra game_toc argument
on the stack could cause ABI issues. The direct call approach (used currently)
is actually correct because non-ToyPad calls don't go through the trampoline —
they're made directly from C. But this needs verification.

STATUS: NEARLY CORRECT — TWO BUGS REMAIN
  1. inject-sprx.js parses old IPC format (INIT_ADDR/INIT_WRAP) but SPRX
     writes new IPC format (TRAMP_BASE/TRAMP_INIT).
  2. PC IP is hardcoded — no mechanism for dynamic configuration.

3. CRITICAL BUG #1: inject-sprx.js IPC Format Mismatch
================================================================================

SEVERITY: BLOCKER — Hooks will never be installed.

ROOT CAUSE:
  usb_hooks.c (refactored 2026-07-22) writes IPC file with keys:
    STATUS=ready
    TRAMP_BASE=0xXXXXXXXX
    TRAMP_INIT=0xXXXXXXXX
    TRAMP_OPENPIPE=0xXXXXXXXX
    TRAMP_TRANSFER=0xXXXXXXXX
    TRAMP_CLOSEPIPE=0xXXXXXXXX

  But inject-sprx.js (lines 381-396) reads the OLD format:
    const targets = [
      { name: 'cellUsbdInit',         target: kv.INIT_ADDR,         wrapper: kv.INIT_WRAP },
      { name: 'cellUsbdOpenPipe',     target: kv.OPENPIPE_ADDR,     wrapper: kv.OPENPIPE_WRAP },
      { name: 'cellUsbdTransfer',     target: kv.TRANSFER_ADDR,     wrapper: kv.TRANSFER_WRAP },
      { name: 'cellUsbdClosePipe',    target: kv.CLOSEPIPE_ADDR,    wrapper: kv.CLOSEPIPE_WRAP },
      { name: 'cellUsbdRegisterLdd',  target: kv.REGISTERLDD_ADDR,  wrapper: kv.REGISTERLDD_WRAP },
    ];

  Result: kv.INIT_ADDR is undefined → preamble skipped for ALL 5 targets.
  Game runs normally but NO hooks installed. No crash, just silence.

  NOTE: Also, the old format had separate TARGET and WRAPPER addresses.
  The NEW architecture uses a SINGLE trampoline address (no separate wrapper).
  The preamble should branch DIRECTLY to the trampoline address.

MIGRATION:
  OLD (Phase 1):  4-instruction @ target -> wrapper -> assembly -> C hook
  NEW (Phase 2):  4-instruction @ game .text -> trampoline -> C hook
                   (Wrapper no longer exists — trampoline IS the wrapper)

FIX:
  See Section 13 for the complete corrected code.

5. RISK #1: inject-sprx.js Writes Directly to R-X .text
================================================================================

SEVERITY: HIGH — Wrong address = instant crash.

DESCRIPTION:
  inject-sprx.js constructs MEMORY SET endpoints with hex data and sends
  them to PS3MAPI for each preamble. The PS3MAPI kernel module bypasses
  the R-X page protection on game .text.

  CURRENT BEHAVIOR (after format fix):
    For each target function address + trampoline address:
      1. Build lis/ori/mtctr/bctr preamble for trampoline address
      2. Write 16 bytes to target function address via MEMORY SET
      3. The game's first 4 function instructions are destroyed

  PROBLEM: The target addresses (TRAMP_INIT, TRAMP_OPENPIPE, etc.) that
  usb_hooks.c writes to the IPC file are the TRAMPOLINE addresses within
  the SPRX's R-W-X page — NOT the game function addresses!

  THE GAME FUNCTION ADDRESSES ARE STILL NEEDED but never written to the IPC
  file. The SPRX used to extract them via OPD trick or NID scan and write
  them as TARGET_ADDR=xxx. That logic was REMOVED in the refactor.

  Without knowing where cellUsbdInit, cellUsbdOpenPipe, etc. live in the
  game's memory, inject-sprx.js cannot know WHERE to write the preamble.

MITIGATION:
  The NID-to-address resolution was previously done by find_cellusbd_functions
  via OPD extraction. But the OPD trick only works for OUR imports, not the
  game's symbols. We need either:
    (A) Re-add NID scanning in the SPRX to find game function addresses,
        and write TARGET_* addresses to the IPC file.
    (B) Have the SPRX itself write the preambles directly to game memory
        (requires the game to have our SPRX mapped at a known address and
        the SPRX having write access to game .text — not guaranteed).
    (C) Have inject-sprx.js use PS3MAPI to scan for the NID patterns in
        game memory, which it already has the ability to do.

  Option (C) is safest — inject-sprx.js already has HTTP access to the
  PS3 and can read game memory. We can have it:
    1. Read the game's process info
    2. Search for known NID patterns (cellUsbdInit, etc.) in the game's .text
    3. Derive the function addresses
    4. Write preambles

  But this is complex. The SIMPLEST option for now:
    (D) Have usb_hooks.c write both TRAMP_* (trampoline addresses) AND
        TARGET_* (game function addresses, obtained by re-adding the NID
        scan or OPD trick) to the IPC file.

  I recommend option (D) — re-add a simple NID scan for the 4 cellUsbd
  functions. The code for this was previously in the codebase.

CURRENT STATUS OF THIS RISK:
  The NID scanning code was removed during the refactor. Without it,
  inject-sprx.js doesn't know WHERE to write preambles to. This is a
  BLOCKER that must be resolved before the hooks can work on real hardware.

6. RISK #2: sys_memory_allocate R-W-X Page Behavior Unknown
================================================================================

SEVERITY: MEDIUM — The trampoline code is correct but untested.

DESCRIPTION:
  usb_hooks.c calls sys_memory_allocate(64KB, SYS_MEMORY_PAGE_SIZE_64K, &addr)
  to get an executable page. The Sony SDK -mprx build environment should
  support this call, but:
    - SYS_MEMORY_PAGE_SIZE_64K might not be defined in SDK 3.40 headers
    - sys_memory_allocate might require a valid container ID (we use 0xFFFFFFFF)
    - The resulting page might be R-W only (not executable) if the PPU's
      page tables don't have the X bit set

  From the PS3SDK documentation, sys_memory_allocate DOES return executable
  pages for PRX-loaded modules. PS3MAPI's MODULE LOAD triggers the same
  path. This is likely safe.

TEST:
  If the SPRX loads but all trampoline calls crash with ISI, the page is
  likely not executable. Alternative: use sys_mmapper (the syscall-level
  memory mapper that handles X bit explicitly). sys_mmapper requires:
    syscall 53 (sys_mmapper_allocate_address)
    syscall 54 (sys_mmapper_allocate_memory)
    syscall 55 (sys_mmapper_map_memory)

  These are in syscall.h and were previously implemented. If the simpler
  sys_memory_allocate fails, fall back to sys_mmapper.

7. RISK #3: sys_net vs sceNet NpTrophy Deadlock
================================================================================

SEVERITY: LOW — Mitigated by SO_NBIO + polling loop.

DESCRIPTION:
  The worker thread calls recvfrom in non-blocking mode (SO_NBIO) on a
  dedicated socket. This means recvfrom returns immediately with -1/
  EWOULDBLOCK when no data is available, then yields for 50ms. The
  sceNet NpTrophy synchronization path is never blocked.

  The main loop explicitly avoids blocking calls:
    - recvfrom via SO_NBIO
    - sendto is inherently non-blocking for UDP
    - sys_timer_usleep yields the hardware thread

  This pattern has been validated by other PS3 homebrew projects.
  RISK LEVEL: LOW.

8. RISK #4: Keepalive Registration Race Condition
================================================================================

SEVERITY: LOW — Graceful handling already in server.js.

DESCRIPTION:
  The SPRX sends keepalive heartbeats (NET_PACKET_TYPE_KEEPALIVE, 0xEE)
  every 3 seconds. The server registers the client on first keepalive.

  Race: If the server receives a USB poll packet (type 0x01) before the
  keepalive, the packet is processed but the client isn't registered yet.
  The server.js handles this gracefully — processPacket() checks for
  client address and routes the response back to the source.

  RISK LEVEL: LOW.

10. RISK #6: Game TOC Passthrough Not Yet Implemented
================================================================================

SEVERITY: MEDIUM — If the game needs real cellUsbd calls to work.

DESCRIPTION:
  The current usb_hooks.c passes through non-ToyPad USB traffic via
  direct calls to cellUsbd* functions:
    return cellUsbdOpenPipe(pipe_handle, dev_id, ep_descriptor);

  This works because the SPRX has its own resolved imports from
  -lusbd_stub. The compiler uses the SPRX's GOT/TOC, and the call
  goes to the REAL cellUsbd implementation in libusbd.sprx.

  However, there is a subtle issue: the game_toc argument is available
  (passed by the trampoline). If there's ever a need to call a function
  in the GAME's address space (not libusbd), we would need to construct
  a fake OPD on the stack using the game's TOC, like this:

    ppc_opd_t fake_opd;
    fake_opd.code_addr = saved_game_code_ptr;  // raw code ptr from GOT
    fake_opd.toc_addr  = game_toc;              // from trampoline arg
    fake_opd.env_ptr   = 0;
    int (*orig_func)(uint32_t*, uint32_t, void*) = (void*)&fake_opd;
    return orig_func(pipe, dev_id, ep);

  This is NOT needed currently because:
    - The real cellUsbd* functions are in libusbd.sprx (system PRX),
      not in the game's own .text.
    - Calling them via the SPRX's import is equivalent to calling them
      via the game's GOT — they resolve to the same libusbd code.
    - The game's TOC is not needed to call system library functions.

  HOWEVER: If we later need to call a function that IS in the game's
  own .text (e.g., a game-specific callback), we'll need the fake OPD
  approach. The game_toc argument is already passed by the trampoline
  for this purpose.

  RISK LEVEL: LOW (no current need). But the infrastructure is ready.

13. REMEDIATION: inject-sprx.js IPC Format Fix (FULL CODE)
================================================================================

Replace lines 367-448 of inject-sprx.js with this corrected preamble writer:

--------------------------------------------------------------------------------
  // ── Parse IPC content into key-value map ──
  const kv = {};
  for (const line of ipcContent.split('\n')) {
    const trimmed = line.trim();
    if (!trimmed || trimmed.startsWith('#')) continue;
    const eqIdx = trimmed.indexOf('=');
    if (eqIdx === -1) continue;
    const key = trimmed.substring(0, eqIdx).trim();
    const val = trimmed.substring(eqIdx + 1).trim();
    kv[key] = val;
  }

  verbose(`IPC parsed: ${JSON.stringify(kv, null, 2)}`);

  // Show discovered trampoline addresses
  log(`  Trampoline base:  ${kv.TRAMP_BASE || 'unknown'}`);
  log(`  Tramp INIT:       ${kv.TRAMP_INIT || 'unknown'}`);
  log(`  Tramp OPENPIPE:   ${kv.TRAMP_OPENPIPE || 'unknown'}`);
  log(`  Tramp TRANSFER:   ${kv.TRAMP_TRANSFER || 'unknown'}`);
  log(`  Tramp CLOSEPIPE:  ${kv.TRAMP_CLOSEPIPE || 'unknown'}`);

  // Build and install preambles for each trampoline.
  // Each preamble is a 4-instruction absolute branch to the trampoline:
  //   [0] lis  r11, hi16(tramp_addr)      3D60xxxx
  //   [1] ori  r11, r11, lo16(tramp_addr) 616Bxxxx
  //   [2] mtctr r11                       7D6B03A6
  //   [3] bctr                            4E800420
  //
  // NOTE: The TARGET function address (where to write the preamble) is NOT
  // yet available from the IPC file. The usb_hooks.c refactor removed the
  // NID scanning that resolved game function addresses. Until that's
  // re-added, we read TARGET_* from a SEPARATE lookup or pass them as
  // command-line arguments.
  //
  // WORKAROUND for Phase 2 development:
  //   The NID-to-address resolution needs to be re-added to usb_hooks.c
  //   or done by inject-sprx.js via PS3MAPI memory scanning. For now,
  //   we support command-line overrides:
  //     --target-init <addr>
  //     --target-openpipe <addr>
  //     --target-transfer <addr>
  //     --target-close <addr>

  function getTargetAddress(name) {
    const cliFlag = `--target-${name.toLowerCase()}`;
    const idx = process.argv.indexOf(cliFlag);
    if (idx !== -1 && process.argv[idx + 1]) {
      return parseInt(process.argv[idx + 1], 16);
    }
    // Fallback: use the TARGET_* key from IPC (if re-added to usb_hooks.c)
    const ipcKey = `TARGET_${name.toUpperCase()}`;
    if (kv[ipcKey] && kv[ipcKey] !== '0x0') {
      return parseInt(kv[ipcKey], 16);
    }
    return null;
  }

  const trampolines = [
    { name: 'cellUsbdInit',        trampKey: 'TRAMP_INIT' },
    { name: 'cellUsbdOpenPipe',    trampKey: 'TRAMP_OPENPIPE' },
    { name: 'cellUsbdInterruptTransfer', trampKey: 'TRAMP_TRANSFER' },
    { name: 'cellUsbdClosePipe',   trampKey: 'TRAMP_CLOSEPIPE' },
  ];

  for (const t of trampolines) {
    const trampAddrStr = kv[t.trampKey];
    const trampAddr = trampAddrStr ? parseInt(trampAddrStr, 16) : NaN;
    const targetAddr = getTargetAddress(t.name);

    if (isNaN(trampAddr) || trampAddr === 0) {
      log(`  ⚠ Skipping ${t.name}: missing trampoline address`);
      continue;
    }

    if (!targetAddr || isNaN(targetAddr) || targetAddr === 0) {
      log(`  ⚠ Skipping ${t.name}: no target function address`);
      log(`      Provide --target-${t.name.toLowerCase()} <hex_addr>`);
      continue;
    }

    log(`  Installing preamble on ${t.name} @ 0x${targetAddr.toString(16)}` +
        ` -> trampoline 0x${trampAddr.toString(16)}`);

    // Build 4-instruction preamble
    const preamble = Buffer.alloc(16);
    preamble.writeUInt32BE((0x3D60 << 16) | ((trampAddr >> 16) & 0xFFFF), 0);  // lis r11
    preamble.writeUInt32BE((0x616B << 16) | (trampAddr & 0xFFFF), 4);          // ori r11,r11
    preamble.writeUInt32BE(0x7D6B03A6, 8);  // mtctr r11
    preamble.writeUInt32BE(0x4E800420, 12); // bctr

    // Write via PS3MAPI MEMORY SET
    const hexData = preamble.toString('hex').toUpperCase();
    const pidHex = '0x' + gamePid.toString(16);
    const addrHex = '0x' + targetAddr.toString(16);
    const writeEndpoint = `/ps3mapi.ps3?MEMORY%20SET%20${pidHex}%20${addrHex}%20${hexData}`;

    verbose(`  Write endpoint: ${writeEndpoint}`);

    try {
      const writeResp = await ps3mapiRequest(writeEndpoint, 5000);
      verbose(`  Write response: ${writeResp ? writeResp.trim() : '(empty)'}`);
      log(`  ✓ Preamble installed on ${t.name}`);
    } catch (err) {
      log(`  ✗ Failed to write preamble on ${t.name}: ${err.message}`);
    }

    await sleep(100);
  }

  log('✓ All preambles installed (where target addresses were available)!');
  log('  Game will now route Toy Pad USB traffic via hooks.');
  return true;

15. REMEDIATION: PASSTHROUGH via Fake OPD (FULL CODE REFERENCE)
================================================================================

While the current direct-call approach works for libusbd system functions,
here is the complete fake-OPD passthrough mechanism for future use when
calling functions in the game's own address space.

This code goes in usb_hooks.c if needed in the future:

--------------------------------------------------------------------------------
/* ── PSEUDO-OPD PASSTHROUGH ──────────────────────────────────────────────
 * Calling a function in the GAME's address space requires the game's TOC.
 * We construct a fake OPD on the stack with the game's code pointer and
 * TOC, then cast it to a function pointer. The C compiler generates the
 * proper r2-switching ABI call sequence.
 *
 * This is ONLY needed if we need to call a function that lives in the
 * game's own .text (not in libusbd.sprx or other system PRXes).
 * Currently NOT used — kept for reference.
 *
 * HOW TO SAVE THE GAME'S GOT SLOT VALUE:
 *   Before overwriting the GOT slot with our trampoline address, the
 *   SPRX reads the original GOT value that the game's PLT stub loaded.
 *   This is a RAW CODE POINTER (not an OPD!) that points to the OS's
 *   TOC-swapping stub for the real cellUsbd* function.
 *
 *   The game's TOC is passed as game_toc argument by the trampoline.
 *   Combining them gives us a valid execution path through the real OS.
 *
 * HOW TO USE (example for cellUsbdOpenPipe):
 *
 *   // Save these at hook init time, before overwriting game's GOT:
 *   static uint32_t g_saved_game_code_ptr_OpenPipe;
 *   g_saved_game_code_ptr_OpenPipe = *game_got_slot_OpenPipe;
 *
 *   // In the passthrough path:
 *   int my_cellUsbdOpenPipe_passthrough(uint32_t *pipe, uint32_t dev_id,
 *                                        void *ep, uint32_t game_toc) {
 *       // Non-ToyPad: call real cellUsbd via game's OS stub
 *       ppc_opd_t fake_opd;
 *       fake_opd.code_addr = g_saved_game_code_ptr_OpenPipe;
 *       fake_opd.toc_addr  = game_toc;
 *       fake_opd.env_ptr   = 0;
 *
 *       int (*orig_func)(uint32_t*, uint32_t, void*) = (void*)&fake_opd;
 *       return orig_func(pipe, dev_id, ep);
 *   }
 *
 * DETAILED EXPLANATION:
 *   On CellOS, function pointers point to 12-byte OPD structs:
 *     { code_addr (4B), toc_addr (4B), env_ptr (4B) }
 *   When you call a function pointer in C, the compiler:
 *     1. Loads the OPD address into r11
 *     2. Loads code_addr from OPD into r12
 *     3. Loads toc_addr from OPD into r2
 *     4. Branches to r12 (the true code entry point)
 *
 *   By constructing our own OPD with:
 *     - code_addr = the OS export stub address (saved from GOT)
 *     - toc_addr  = the game's TOC (passed by trampoline)
 *   We create a valid call path that enters the OS's TOC-switching
 *   stub with the game's TOC, which correctly handles the transition
 *   to the OS PRX's address space.
 *
 *   The env_ptr is always 0 (unused on CellOS 32-bit).
 */

17. APPENDIX: PowerPC Instruction Reference for Debugging ISI Crashes
================================================================================

If the PS3 crashes with an Instruction Storage Interrupt (ISI), the CPU
tried to execute data at an invalid or non-executable address. Here's how
to decode what happened:

COMMON ISI CAUSES IN THIS SYSTEM:

CAUSE 1: GOT slot contains an OPD address instead of a code address
  Symptom: ISI at the OPD address (where code_addr/toc_addr/env_ptr are stored)
  Why: The PS3 PLT stub loaded the GOT slot value and branched to it via
       bctr. If the value is an OPD pointer, the CPU executes OPD bytes.
  Fix: Ensure the GOT slot contains the trampoline ADDRESS, not a pointer
       to our C function (which is an OPD).

CAUSE 2: Trampoline page is not executable
  Symptom: ISI at the trampoline base address
  Why: sys_memory_allocate returned an R-W page without X bit
  Fix: Use sys_mmapper to explicitly set X permissions

CAUSE 3: Cache was not flushed after writing trampoline
  Symptom: ISI with instruction = 0x00000000 (stale data cache line)
  Why: trampoline_gen.c writes via data stores (stw), but instruction
       cache still has old zeros at that address
  Fix: dcbst/sync/icbi/isync per instruction word (ALREADY IMPLEMENTED)

POWERPC OPCODES FOR MANUAL DEBUG:
  0x9421FFA0   stwu r1, -0x60(r1)
  0x7C0802A6   mflr r0
  0x90010064   stw r0, 0x64(r1)
